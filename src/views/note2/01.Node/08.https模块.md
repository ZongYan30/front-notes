# https模块

> http + SSL加密 默认端口是443

加密的演变：对称加密-->非对称加密-->CA证书

加密的流程比较抽象，带入三个角色：
monica
邓哥
黑客(成哥)

## 对称加密

1. 邓哥 和 monica 约定最开始的时候使用一个公钥加密对话信息，双方都拥有一个公钥
2. 邓哥--> 使用公钥加密对话"我喜欢你" --> 发送给monica
3. monica --> 使用公钥解密 --> 得到"我喜欢你"

**优点：**初步对信息进行了加密

**弊端：**当黑客在中间拦截的时候，得到公钥信息，可以篡改邓哥发送的内容。

## 非对称加密

1. monica-->生成一对公钥key1和私钥key1-->发送公钥key1给邓哥。
2. 邓哥-->生成私钥key2-->使用key1将key2加密-->发送给monica
3. 邓哥-->使用key2发送我爱你-->monica
4. monica-->使用key2解密-->读取内容-->使用key2加密你是好人-->邓哥

<img src="https://zongyan30.github.io/front-assets/Node/https02.png"/>

当黑客出现劫持:

- 黑客可以在最初的时候劫持到monica发送的公钥，然后自己生成自己一对假的公钥和私钥
- 黑客-->发送假公钥给邓哥
- 邓哥-->发送被假公钥加密的key2给黑客-->篡改信息为我恨你，用key2加密-->monica

## https-CA证书

为了解决上面的问题，需要一个第三方权威机构来做中间担保人。

- 浏览器内部都内置的有对应的权威机构，并且有其对应的公钥。
- 百度公司将自己的主机、公司信息、公钥发送给CA机构，由机构生成对应的证书。
- 那么，该证书需要使用百度的公钥进行解密，如果中间任何一个信息被篡改了，那么生成对应的证书就不一样，浏览器直接会报出警告。

流程如下：

1. 百度服务器-生成公钥key1/私钥key1 --自己保留私钥 --申请CA证书，将公钥发送给CA机构。
2. 浏览器验证CA证书--> 通过后使用百度公钥加密自己生成的私钥key2-->发送给百度服务器
3. 百度使用私钥key1解密信息--得到私钥key2

黑客此时就无法篡改任何内容了，一旦劫持到证书，更改其中的任何一项信息，那么浏览器侧解析得到的证书都不一致。

## https模块代码

https一般需要收费，具体可见主流供应商收费规则。
开发中常常使用http模块进行开发，最后使用nginx服务器进行代理，接入https。

```js
const https = require("https");

const server = https.createServer(
  {
    key: fs.readFileSync(path.resolve(__dirname, "./server-key.pem")), //私钥
    cert: fs.readFileSync(path.resolve(__dirname, "./server-cert.crt")), //证书
  },
  (req, res) => {
    console.log("请求来了", req, res);
  },
);

server.on("listening", () => {
  console.log("server listen 443");
});

server.listen(443);
```
